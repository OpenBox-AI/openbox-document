# Getting Started with OpenClaw

OpenBox integrates with [OpenClaw](https://openclaw.dev) as a plugin — every tool call and LLM inference your agent makes is evaluated against your OpenBox policies before execution.

The plugin provides two governance paths:
- **Tool governance** — intercepts every tool call via `before_tool_call` / `after_tool_call` hooks
- **LLM guardrails** — routes LLM requests through a local gateway for PII detection, content filtering, and more

## Prerequisites

- **Node.js 22+**
- **OpenClaw** installed and running
- **OpenBox API Key** — [Register your agent](/dashboard/agents/registering-agents) in the dashboard to get one

You will need two values from your OpenBox account:

| Value | Format | Where to find it |
|-------|--------|------------------|
| OpenBox URL | `https://core.openbox.ai` | Your OpenBox Core instance URL |
| API Key | `obx_live_*` or `obx_test_*` | Settings > API Keys in the OpenBox dashboard |

## Step 1: Build and Install the Plugin

```bash
cd /path/to/openbox-openclaw-plugin
npm install
npm run build
```

Install the plugin into OpenClaw:

```bash
openclaw plugins install .
```

## Step 2: Configure the Plugin

Add your OpenBox credentials to your `.env` file in OpenClaw:

```bash title=".env"
OPENBOX_URL=https://core.openbox.ai
OPENBOX_API_KEY=obx_live_your_key_here
OPENAI_API_KEY=sk-your_openai_key_here  # your existing LLM API key
```

Then add the plugin config to your `openclaw.json`:

```json title="openclaw.json"
{
  "plugins": {
    "entries": {
      "openbox": {
        "enabled": true,
        "config": {
          "openboxUrl": "${OPENBOX_URL}",
          "openboxApiKey": "${OPENBOX_API_KEY}",
          "llmBaseUrl": "https://api.openai.com/v1",
          "llmApiKey": "${OPENAI_API_KEY}",
          "gatewayPort": 18919
        }
      }
    }
  }
}
```

You can also configure these values from the OpenClaw web control UI instead of editing files directly.

## Step 3: Point Your LLM Provider to the Gateway

To enable LLM guardrails (PII detection, content policy), route your model provider through the OpenBox gateway. In your `openclaw.json`, update the model provider's `baseUrl`:

```json title="openclaw.json"
{
  "models": {
    "providers": {
      "openai": {
        "baseUrl": "http://127.0.0.1:18919/v1",
        "apiKey": "${OPENAI_API_KEY}",
        "api": "openai-responses",
        "models": [
          {
            "id": "openai/gpt-5-mini",
            "name": "openai/gpt-5-mini"
          }
        ]
      }
    }
  }
}
```

The model choice is flexible — any OpenAI-compatible model and provider will work.

:::info Tool governance works independently
If you skip the gateway step, tool governance still works — policies are enforced on every tool call. Only LLM guardrails (PII redaction, content filtering) require the gateway.
:::

## Step 4: Restart the Gateway

```bash
openclaw gateway restart
```

The gateway must be restarted after any plugin or configuration change.

## Step 5: Verify Your Setup

After restarting, check the OpenClaw logs for:

```
[openbox] plugin loaded
[openbox] LLM gateway started on http://127.0.0.1:18919/v1
```

If you configured the LLM gateway, verify the health endpoint:

```bash
curl http://127.0.0.1:18919/health
# Expected: {"status":"ok"}
```

Run a tool call and check for governance output:

```
[openbox] session registered verdict=allow
[openbox] tool=Read verdict=allow reason="" ms=120
```

This confirms the plugin is intercepting tool calls, OpenBox Core is reachable, and your API key is valid.

For a full verification checklist, see **[Verifying Your Setup](/developer-guide/openclaw/troubleshooting#verifying-your-setup)**.
