# Getting Started with Cursor Hooks

OpenBox governance for [Cursor IDE](https://cursor.com) via the official [hooks system](https://docs.cursor.com/context/hooks). Every agent action — prompts, shell commands, MCP tool calls, file reads — is governed through OpenBox before it executes.

## How it works

Cursor's hooks system fires custom scripts at every point in the agent loop. `cursor-hooks` registers a single handler that routes each hook event through OpenBox's governance engine.

```
User types prompt
  → beforeSubmitPrompt → OpenBox: ALLOW or BLOCK

Agent reads file
  → beforeReadFile → OpenBox: ALLOW, BLOCK, or CONSTRAIN (redact secrets)

Agent runs shell command
  → beforeShellExecution → OpenBox: ALLOW or BLOCK

Agent calls MCP tool
  → beforeMCPExecution → OpenBox: ALLOW or BLOCK

MCP tool returns data
  → afterMCPExecution → OpenBox: redact PII from response

Agent responds
  → afterAgentResponse → OpenBox: observe, update trust score

Session ends
  → stop → OpenBox: close workflow, finalize audit trail
```

## Prerequisites

- [Node.js](https://nodejs.org/) v18+
- [Cursor IDE](https://cursor.com) v2.5+
- An OpenBox API key (`obx_live_xxx`) — get one at [platform.openbox.ai](https://platform.openbox.ai)

## Quick start

### 1. Clone and install

```bash
git clone https://github.com/OpenBox-AI/cursor-hooks.git
cd cursor-hooks
npm install
npm run build
```

### 2. Install hooks

```bash
npm run install-hooks -- --key obx_live_your_key
```

This does three things:
- Creates `~/.cursor-hooks/config.json` with your API key
- Creates a stable symlink at `~/.cursor-hooks/hook-handler.js`
- Installs all 11 hooks into `~/.cursor/hooks.json`

### 3. Restart Cursor

Quit Cursor completely (`Cmd+Q`) and reopen it. All agent actions are now governed.

### 4. Verify

Open a terminal and tail the log:

```bash
tail -f ~/.cursor-hooks/hook.log
```

Send any prompt in Cursor — you should see hook events appear in the log.

## Installation options

```bash
# Global (recommended) — applies to all Cursor projects
npm run install-hooks -- --key obx_live_your_key

# Project-level — only applies to the current workspace
npm run install-hooks -- --key obx_live_your_key --project

# Dry-run mode — allows all actions, skips API calls
npm run install-hooks -- --key obx_live_your_key --dry-run
```

## Configuration

Config file: `~/.cursor-hooks/config.json`

| Setting | Required | Default | Description |
|---------|----------|---------|-------------|
| `OPENBOX_API_KEY` | Yes | — | API key (`obx_live_xxx`) |
| `OPENBOX_ENDPOINT` | No | `https://core.openbox.ai` | Core API URL |
| `GOVERNANCE_POLICY` | No | `fail_open` | `fail_open` or `fail_closed` |
| `GOVERNANCE_TIMEOUT` | No | `15` | API timeout (seconds) |
| `VERBOSE` | No | `false` | Debug logging to stderr |
| `DRY_RUN` | No | `false` | Allow all, skip API |

## What gets governed

| Hook | Can block? | Activity type |
|------|-----------|---------------|
| `beforeSubmitPrompt` | Yes | `PromptSubmission` |
| `beforeReadFile` | Yes (+ redact) | `FileRead` |
| `beforeShellExecution` | Yes | `ShellExecution` |
| `beforeMCPExecution` | Yes | `MCPToolCall` |
| `afterMCPExecution` | Redact only | `MCPToolResponse` |
| `afterAgentResponse` | No | `AgentResponse` |
| `afterAgentThought` | No | `AgentThinking` |
| `afterShellExecution` | No | `ShellOutput` |
| `afterFileEdit` | No | `FileEdit` |
| `sessionStart` / `stop` | No | `CursorSession` |

## Verdict mapping

| OpenBox verdict | Cursor behavior |
|----------------|----------------|
| **ALLOW** | Action proceeds |
| **CONSTRAIN** | Content redacted (files, MCP responses) |
| **REQUIRE_APPROVAL** | Blocked. Approve on OpenBox dashboard, then retry. |
| **BLOCK** | Blocked with policy/guardrail message |
| **HALT** | Blocked, session terminated |

## Dashboard setup

Configure guardrails and policies per activity type in **Agent → Authorize**:

| Activity type | What to configure |
|--------------|-------------------|
| `PromptSubmission` | PII, toxicity, ban words, content filter |
| `FileRead` | PII/secret redaction |
| `ShellExecution` | Rego policies (command allowlists) |
| `MCPToolCall` | Rego policies, tool allowlists |
| `MCPToolResponse` | PII redaction on output |

## Logs

```bash
tail -f ~/.cursor-hooks/hook.log
```

Verbose output also appears in Cursor's **View → Output → Hooks** channel when `VERBOSE` is enabled.

## Uninstall

```bash
rm ~/.cursor/hooks.json
rm -rf ~/.cursor-hooks
```
